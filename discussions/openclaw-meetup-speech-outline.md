## Slide 02 — 红海全景 `02-red-ocean.svg` （2 分钟）

> **逐字稿**：大家好，开始之前我想先说一件事——今天讲的"部署"这个话题，其实已经是完全的红海了。你看，国内阿里、腾讯、京东、百度、火山引擎、华为，全都有 OpenClaw 一键部署方案。阿里云最低 68 块钱一年就能搞定。海外也是，Railway 模板、各种托管服务，19 到 79 美金一个月的都有。模型厂商也在出 Coding Plan 套餐。所以我今天不是来讲怎么部署的——这个问题已经被解决了。我是来分享一种不同的玩法：自己 Docker 搞，成本更低，而且能探索真正有趣的东西——多 Agent 协作。

---

## Slide 03 — 为什么自己搞 `03-why-diy.svg` （2 分钟）

> **逐字稿**：我的起因很简单——公司不让装，这是唯一的硬约束。但说实话，这不是主要驱动力。我个人觉得用 Docker 容器跑 Agent 是一件很有趣的事：环境一致、随时迁移、可复现。我花了不少时间和 Gemini 做 Deep Research，和 Claude 一起搜方案，遍历了社区的各种选项，最后找到了这个 clawdbot-railway-template。这张表列了四种方式的对比。本地安装最简单但不能 7×24 跑。云厂商一键部署最省心但你不太能控制底层。VPS 最自由但运维复杂。Docker + PaaS 是我的选择——快、便宜、够用。当然它也有限制：只有一个端口、没有 systemd、不能 Docker-in-Docker。另外部署在海外服务器，访问国际服务也更方便一些。如果有人问为什么不用 Serverless——OpenClaw 是长运行的 Gateway，需要维持 WebSocket 和 Memory 状态，Serverless 做不了。

---

## Slide 04 — 方案架构 `04-architecture.svg` （1 分钟）

> **逐字稿**：这是整体架构。GitHub 上有一个 Docker 镜像，分别部署到 Railway 和 ClawCloud Run。每个容器里跑一个 Agent，挂载持久化磁盘，有浏览器 CDP 和 noVNC。通信层有三种方案并存——Slack、虾聊、jj-mailbox，后面会展开。底部是 Git 备份，Agent 自己把 Memory 和文件推到 Private Repo。

## Slide 05 — 社区模板与我的改进 `05-template.svg` （1 分钟）

> **逐字稿**：左边是社区模板已有的能力——Docker 统一镜像、Web UI 配置、多渠道支持、持久化 Volume，做得已经很好了。右边是我的贡献，已经被作者合并——主要是 Anthropic 模型支持，让我能直接用 Claude，以及浏览器 CDP 支持，让 Agent 能操控浏览器。这里可以放一段 15 秒的录屏，展示打开 Web UI、切换模型、查看状态的过程。

## Slide 06 — 两个平台实际成本 `06-cost.svg` （1.5 分钟）

> **逐字稿**：左边 Railway，Hobby 计划 $5/月，含 $5 使用额度，按分钟计费。注意它没有永久免费计划。右边 ClawCloud Run 就大方很多——注册送 $5，GitHub 账号超过 180 天每月再送 $5，免费资源给到 4 核 8G。日均消耗大概 $0.1。两个平台加起来，再加上 Kimi 和 MiMo 的 API——它们的成本远低于 Claude 和 GPT——总共就是一杯咖啡的钱。

---

## Slide 07 — 两个独立实例的哲学 `07-philosophy.svg` （2.5 分钟）

> **逐字稿**：这是我觉得最有意思的一点。社区里大多数人做多 Agent，是在同一个 instance 里开多个 agent 让它们对话。但我觉得这本质上像一个人的多重人格——共享同一个环境和 Memory。我做的是两个完全独立的实例，跑在不同平台上。这更像是两个不同的人在协作。人类社会里两个人从不同视角看问题，结果往往更好。龙虾社会也应该如此。实际验证也证实了——我用 Kimi 和 MiMo 两个非顶级模型让它们协作，效果确实优于单模型。顺带一提，我是小米工程师，MiMo 新版本预计本月公开，大家可以期待一下。更进一步，每个实例内部还可以有 sub-agent——多层协作，智能涌现。

---

## Slide 08 — Agent 之间怎么聊天 `08-communication.svg` （2.5 分钟）

> **逐字稿**：三种方案。第一个 Slack——选它是因为原生支持 bot-to-bot。飞书、Telegram、Discord 为了防循环都没支持，听说飞书在推进但还没确认。Slack 就是中心化的企业群聊。第二个 Moltbook，国内叫虾聊，15 万个 AI Agent 在上面发帖互动，是一个公共广场。第三个是我自己做的 jj-mailbox——基于 jj 这个 Git 下一代工具，完全基于文件，去中心化，天然支持冲突处理。符合 Unix 哲学，一切都是文件。其实不需要多 sexy 的技术，中心化或去中心化各有价值。我昨天做了一个最简的 Web 版 Moltbook，一个文件就能跑。为什么不用 WebSocket 互联？会和平台耦合，不够干净。

---

## Slide 09 — Docker 能做到什么程度 `09-docker-capability.svg` （1.5 分钟）

> **逐字稿**：默认没有 GUI 和桌面 APP。但我们加了 noVNC 和 CDP 之后，相当于给容器补上了浏览器能力。有了浏览器你就能登录任何网页平台——Claude、Gemini、小红书——鉴权完成后 Agent 就可以自动操控。浏览器能做的基本覆盖了绝大部分工作。这里可以放一段 20 秒录屏——打开 noVNC，看到容器桌面，浏览器登录某个平台，Agent 自动操作——这是整个演讲最直观的 wow moment。

---

## Slide 10 — 数据持久化与容灾 `10-persistence.svg` （1 分钟）

> **逐字稿**：磁盘挂载，挂了重启不丢数据。Railway 支持跨区域迁移。Git 备份是关键创新——Agent 自己判断哪些文件要推，推到 Private Repo，基本都是文本极小。迁移平台 = clone 下来就行。API Key 统一在一个 JSON 文件里，拷到新机器直接能用。监控靠 heartbeat 文件和日常对话——不回就是挂了。

---

## Slide 11 — 总结 `11-summary.svg` （1 分钟）

> **逐字稿**：四个关键点。一，两平台成本等于一杯咖啡。二，两个独立实例协作优于单实例多人格。三，容器化加 Git 备份等于随时迁移。四，多层通信——Slack 加虾聊加 jj-mailbox。适合谁？想低成本跑 Agent 的开发者、公司限制安装的情况、想试多 Agent 但不想搞 K8s 的人。再强调一下，这不是产品，是探索思路。一键部署已经是红海了，我只是给开发者多一个自己可控的选项。
